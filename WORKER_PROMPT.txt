PHASE 2 — Implement Worker per contract

File structure per contract §16.2:

src/
├── index.ts              ← entry point, route dispatch
├── auth/
│   ├── jwt.ts            ← JWT verify (HS256, dual-secret support)
│   └── adminAuth.ts      ← X-Backend-Auth verify (constant-time)
├── handlers/
│   ├── upload.ts         ← PUT /{key}
│   ├── view.ts           ← GET /{key} (public + private)
│   ├── head.ts           ← HEAD /{key}
│   └── options.ts        ← CORS preflight
├── lib/
│   ├── cors.ts           ← origin matching, preflight responses
│   ├── errors.ts         ← buildErrorResponse(code, status, details)
│   ├── logger.ts         ← structured JSON via console.log(JSON.stringify)
│   ├── pathValidation.ts ← path traversal rejection
│   ├── rateLimit.ts      ← per-token + per-IP via KV (or skip v1, see below)
│   └── requestId.ts      ← UUID v4 generation/extraction
└── types.ts              ← shared types: JWT claims, error codes, etc.

Implementation requirements per contract:

1. PUT /{key} — validation order EXACTLY as in §4.1 (10 steps). Implement each 
   as a separate function returning Result<T, WorkerError>. Fail fast on any 
   step.
2. GET /{key} — public flow vs private flow per §4.2. Public path 
   (`public/*`): no auth, R2 get, return with cache headers. Private path 
   (`private/*`): token from `?token=` query param, full validation chain.
3. JWT verify — per §5: try JWT_SIGNING_SECRET first, fall back to 
   JWT_SIGNING_SECRET_PREVIOUS if set. Return distinct error 
   (TOKEN_SIGNATURE_INVALID) on signature failure across both.
4. Idempotent retry on consumed token — per contract §4.1 step 10. Worker 
   does R2.head() before writing; if object exists with same content-type 
   and approximately same byte count, return success without rewriting.
5. CORS — per §9. NO `*`, suffix-match for vercel previews, `Vary: Origin` 
   on every CORS-affected response.
6. Logging — per §10. Every code path that ends in a response also emits 
   exactly one log entry with the right level and fields.
7. x-request-id — generate (UUIDv4) if caller didn't provide, echo in EVERY 
   response (success + error), include in every log entry.

Rate limiting (§11.2, §11.3): 

Implement per-token PUT limit (5 attempts) and per-IP PUT limit (100/sec) 
using Cloudflare Workers Durable Objects OR KV with TTL. If implementing 
Durable Objects adds significant complexity, IMPLEMENT v1 WITHOUT 
RATE-LIMITING and document the gap clearly in README. The contract allows 
this — backend gates issuance via Bucket4j already, and the Worker can be 
hardened later.

Decision: I prefer skipping Worker-level rate limiting in v1 if it adds 
non-trivial complexity. State your recommendation and proceed accordingly. 
Document the trade-off either way.

PHASE 3 — Tests

Vitest tests for:

- JWT verification: valid token, expired, wrong signature, wrong issuer, 
  wrong scope, wrong key, wrong contentType
- Path validation: ../, /absolute, %2e%2e, backslashes, empty segments, 
  excessively long keys
- CORS: allowed origin echoed, disallowed origin not echoed, vercel 
  preview wildcard, OPTIONS preflight
- Upload happy path: valid token + valid bytes → 200 with key
- Upload errors: each error code at least once
- View public: existing key → 200, missing → 404
- View private: valid token → 200, expired → 401, key/prefix mismatch → 403
- Idempotent retry: PUT same key twice → both 200, only one R2 write 
  (verify with mock)
- Backend auth: missing header → 401, wrong header → 401 (when admin 
  endpoints exist; document if v1 has none)

Coverage target: 80%+ on handlers and auth modules.
